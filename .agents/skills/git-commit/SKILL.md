# Git Commit

## Overview

Draft one-line commit subjects that match ADHOC or Odoo-style history.

Prefer staged changes. If nothing is staged, inspect the working tree and state that the draft is based on unstaged changes.

Do not create the git commit unless the user explicitly asks for it.

## Workflow

1. Inspect the actual change set.
- Prefer `git diff --cached --name-only`, `git diff --cached --stat`, and `git diff --cached`.
- If nothing is staged, fall back to `git diff --name-only`, `git diff --stat`, and `git diff`.

2. Infer the module or addon name.
- Prefer the directory or addon that contains most of the changed files.
- If one module clearly dominates, use that module name in the subject.
- If the change spans multiple unrelated modules and no single scope dominates, ask the user before drafting.

3. Inspect recent human-written commit history.
- Use recent commit subjects to match local naming and capitalization.
- Ignore autogenerated translation commits whose subject is `Update translation files`.
- Ignore commits whose body mentions Weblate or `msgmerge`.
- Do not use ignored commits as style references.

4. Choose the prefix.
- Use only `[ADD]`, `[IMP]`, `[REF]`, `[I18N]`, `[FIX]`, or `[DEL]`.
- See `references/commit-conventions.md` for selection rules and examples.

5. Draft the subject.
- Format: `[TAG] module_name: concise summary`
- Keep it to one line.
- Keep it specific to the dominant change.
- Do not end with a period.
- Prefer imperative or repository-consistent summary style over generic labels like `changes` or `updates`.
- Prefer English unless the repository clearly uses another language for human-written commits.

## Output

When the user asks for a suggested commit message, return:

```text
[TAG] module_name: summary
```

If the choice of prefix or module is ambiguous, explain the ambiguity briefly and still provide the best draft.

When the user explicitly asks to perform the commit:
- Verify what is staged first.
- Use the drafted subject as the commit message.
- Stop and ask if there are no staged changes and the user did not clearly ask to commit all working tree changes.
