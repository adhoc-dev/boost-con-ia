# Commit Conventions

## Valid Prefixes

- `[ADD]`: Add a new module, feature, model, endpoint, workflow, or other user-visible capability.
- `[IMP]`: Improve existing behavior without the change being primarily a bug fix or a pure refactor.
- `[REF]`: Refactor internal code or structure without intended behavior changes.
- `[I18N]`: Update translations, exported terms, or localization-only content.
- `[FIX]`: Correct broken, wrong, unstable, or regressed behavior.
- `[DEL]`: Remove obsolete code, behavior, fields, modules, or assets.

## Subject Pattern

Use:

```text
[TAG] module_name: summary
```

Guidelines:

- Use the addon or module technical name as `module_name`.
- Keep the summary short and concrete.
- Prefer the dominant behavior change over implementation detail.
- Avoid filler like `misc`, `updates`, `changes`, or `improvements`.
- Do not add a trailing period.

## Examples To Emulate

Good examples:

- `[ADD] ocr_invoice_poc: New Module`
- `[FIX] saas_provider_upgrade: Increase timeout for getting clients in upgrade request`
- `[I18N] saas_client_base_automation: export source terms`
- `[IMP] saas_client_base_automation: skip _check_min_cron_interval on automations`
- `[REF] sale_order_sync: simplify invoice payload builder`
- `[DEL] account_move_legacy: remove deprecated OCR hooks`

## Examples To Ignore

Do not use these as style references:

- `Update translation files`
- Commits generated by Weblate
- Commits whose body says `Updated by "Update PO files to match POT (msgmerge)" add-on in Weblate.`

If the only meaningful change is translation export or import, use `[I18N]`.
If a commit includes both code and translation noise, choose the prefix that matches the code change and ignore the generated translation commit style.
